### memo
#### ネイティヴ型メソッドを使わない方が良い理由
たとえば

cb.query().setMemberStatusCode_Equal("FML")
だと、"FMl" や "FORMAL" のようにタイプミスしても、Java 的にはただの String なのでコンパイルエラーにならない。
検索結果が0件になるだけで気づきにくい。

区分値メソッドなら

cb.query().setMemberStatusCode_Equal_正式会員()
cb.query().setMemberStatusCode_Equal_AsMemberStatus(CDef.MemberStatus.正式会員)
のように書ける

タイポしにくい
名前で意味がわかる
存在しない区分値は生成されない
区分値を削除したときにコンパイルエラーで気づける

区分値を作る前であれば、setMemberStatusCode_Equal("WDL")という書き方ができるが、一度区分値を作ってしまうと、
setMemberStatusCode_Equal("WDL")が使えなくなる
厳密には、存在はするがprotectedメソッドになり、テストクラスなどから呼び出せなくなる

#### 消えるのでなくprotectedメソッドで残るのは何故か
「別アプリで使っている未共有の値や、把握されていない値、いわゆる「隠れ区分値」がある現場ではトラブル原因になりうる」から
dbfluteを導入した時にすでに「setMemberStatusCode_Equal("WDL")」があったら、全部コンパイルエラーになってしまう

#### classificationUndefinedHandlingTypeを設定すると実現されること
- 暗黙の区分値(区分値テーブルでなく、カラムの一部として表現されているもの)に設定外の値が入ってきた時にエラーになる
- 区分値テーブルで区分値として定義されていない値が検索されたときにエラーになる
